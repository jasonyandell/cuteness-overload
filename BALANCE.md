# Cuteness Overload — balance report

Generated by the headless AI harness: `scripts/ai-play.ts` (player) + `scripts/balance.ts`
(runner). Two strategies, all 3 maps × seeds 1–5.

- **saver** — the intended winning play. Keeps a fixed build ORDER (shopping list) and
  only buys the next item when affordable — never buys ahead, never substitutes a pile of
  cheap plinkers for the big towers it is saving toward. Places on the build cell that
  covers the most path (precomputed sample-point coverage), and skips a wave for the bonus
  only when the field is clear and nothing is still spawning. After the fixed list it spends
  the overflow on the best DPS-per-dollar cannon / lightning / doom or upgrade.
- **spender** — the impatient baseline that should lose. Buys a plinker on the best open
  spot the instant it can afford one; once out of spots it pours money into plinker
  upgrades. Never saves for a big tower, never skips.

Run it yourself:

```
npx tsx scripts/balance.ts                 # the whole table
npx tsx scripts/ai-play.ts double 3 saver  # one game, per-wave log
```

Each game sims in ~15–25 ms; the full 30-game sweep finishes in well under a second.

```
double   spender     0/5     17.0       0.0         0
```

Targets: saver wins all → **FAIL** (double), spender loses all → **PASS**.

Meadow and Creek are already well-tuned: the saver wins both but only just (min 2 and 5
lives), and the plinker-spammer loses. **Double Trouble cannot be won at all** — not by the
scripted saver, and not by a stronger reference bot I tried (a pure value-greedy full-spender
that fields ~35 towers still tops out around wave 17). So Double is a genuine tuning problem,
not merely a dumb AI.

### Why Double is unwinnable

It is the two-lane map, and the difficulty is *structural*, not just its `hpMul`:

- Enemies split across two paths, so the same tower budget must defend two fronts at once.
- Path B is only **17.3** world-units long (Meadow's single path is 34.6). A boss on the
  short lane reaches the house in ~30 s and the defense simply cannot concentrate enough
  fire on it in time.

I verified this is the map and not the build: on a losing Double run and a winning Meadow
run the saver fields the *same* ~12 towers with the *same* composition. Dropping Double's
`hpMul` all the way to **1.0** (Meadow's value) still isn't enough — the saver reaches
wave 19 and dies. The short second lane, not raw HP, is the wall.

### The other wall: wave 20 everywhere

Even on Meadow/Creek the saver only scrapes through (min 2 lives on Meadow). Wave 20 sends
**two ~18,000-HP bosses** (`420 × 1.22¹⁹`) plus escorts. `HP_GROWTH = 1.22` compounds to
**44×** by wave 20, which outruns any affordable tower line. The single biggest anti-boss
tool, the Snuggle Nuke, fires only once every 5.5 s (`rate 0.18`), so it barely contributes.

## Recommended changes

I did **not** touch `HP_GROWTH` globally: lowering it does make the saver win Double, but it
also lets the plinker-spammer win Meadow/Creek (everything gets easier equally, erasing the
saver-vs-spender gap). The fixes below are surgical — they help the *premium-tower* saver
and the *hardest map* without helping plinker-spam.

All values were found by copying the sim into a scratch dir and sweeping; the projected table
below is the measured result with exactly these edits.

### 1. `src/sim/constants.ts` — speed up the Snuggle Nuke

```
TOWERS.doom.rate:   0.18  ->  0.26
```

The doom is the only tower whose per-hit damage (90, +2.4 splash) can dent a wave-20 boss,
but at 0.18/s it fires too rarely to matter. 0.26/s (~1 shot per 3.8 s) gives the saver real
boss-killing margin on wave 20. The spender owns **zero** dooms, so this does not help it at
all — it only widens the gap. This alone lifts Meadow/Creek from a nervy min-2 to a
comfortable min-8/9.

### 2. `src/sim/maps.ts` — make Double Trouble winnable

```
double  hpMul:      1.2   ->  1.1
double  incomeMul:  1.1   ->  1.8
```

Keep Double the tankiest map (`hpMul 1.1` is still the highest of the three) so it *reads* as
hardest, but pay the player enough (`incomeMul 1.8`) to afford defending two lanes.
Thematically clean: "two lanes of trouble, double the bounty." The extra income does **not**
rescue the spender — under 1.8× income it fields 83 plinkers and *still* loses at wave 17,
which is the whole point: quantity of cheap single-target towers can't do what a few premium
splash/nuke towers can.

## Per-map difficulty feel (with the recommended changes)

**Meadow Lane** — the tutorial map, and it plays like one. One long, generous path with 87
build cells means the saver always has good coverage and a fat economy. It coasts to ~L9,
never seriously threatened until the wave-20 bosses, which the sped-up doom now handles with
a few lives to spare. The right amount of easy: you can make mistakes and recover.

**Twisty Creek** — a notch harder and it feels it. The path is the longest (45 units) but
`maxBuildDist 1` gives cramped, pocketed build space and `hpMul 1.15`, so every tower slot
matters and the saver can't just spam coverage. Lands around L8–9 — comfortably won, but you
feel the switchbacks funneling enemies past thin spots. A solid mid-difficulty map.

**Double Trouble** — the hardest, and now it earns the name honestly. Two lanes split your
fire and the short second lane barely gives you time to kill a boss before it's home, so the
back half (waves 12–20) is a constant scramble to keep both fronts covered. The 1.8× income
is what makes it *possible* — you're always spending, never hoarding — and even then it
finishes at L6–7. This is the map where the "save for premium AoE, don't spam plinkers"
lesson is life-or-death: the spender, drowning in money, still leaks out at wave 17.

## Known deviation: spender loses at wave 17–20, not the hoped-for 10–16

The spender loses **every** game (primary target met), but it dies late — wave 17 on Double,
wave 20 on Meadow/Creek — rather than collapsing at wave 10–16. This is a real finding, not a
harness bug: a *competently placed* plinker spam (best-coverage cells + upgrades) is more
viable than the design assumed, because an upgraded plinker maxes at ~266 DPS — the same as a
Boop Cannon at 3× the cost. It carries the spender all the way to the final boss wall, where
it finally fails (no splash, no nuke). So the "saving is required" lesson still lands, just at
the finish line instead of mid-game.

I deliberately did **not** force an earlier spender loss, because the only levers that do it
are bad ones:

- Nerfing the plinker (damage/rate) also guts the saver's early game — they share the tower —
  and drops the saver below a winning line. (Tested: `plinker.damage 7→6` makes the saver lose
  Meadow/Creek too.)
- Making the spender place "naively" on arbitrary cells is pathological, not impatient — it
  strands towers where they can't reach the path and dies at wave 4. Not a meaningful baseline.

If hitting the 10–16 window is important, the clean fix is on the **wave side, not the tower
side**: lean the mid-game waves (≈11–16) harder into fast-swarms and shields, which punish
single-target plinker fire specifically while the saver's cannon/lightning/freeze shrug them
off. That widens the gap where the design wants it without weakening either tower. I left
`waves.ts` untouched since the primary targets are met, but that's the lever to pull.
